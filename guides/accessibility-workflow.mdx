Uxia's Accessibility Test is designed for teams that want to catch accessibility issues before development. Instead of waiting for QA, audits, or legal review, you can upload product screens, choose a WCAG target, and get a criterion-by-criterion report with design and code guidance.

## Before you start

Have these ready before you create the test:

- A clear test name
- The screens or recordings you want to audit
- A decision on whether you want to test against **AA** or **AAA**

Supported uploads in the current flow:

- Images: **PNG, JPEG, WEBP**
- Videos: **MP4, MOV, AVI**
- Limit: **up to 10 files**
- Size limit: **100 MB per file**

## Step-by-step setup

<Steps>
  <Step title="Create a new Accessibility Test">
    From **Home** or **Tests**, click **Create test** and choose **Accessibility Test** from the test type modal.
  </Step>
  <Step title="Name the test and optionally assign a project">
    Enter a descriptive title so the run is easy to find later. You can also attach the test to a project, or leave it as **No project** if you only need a standalone audit.
  </Step>
  <Step title="Choose your WCAG conformance level">
    Uxia asks you to select the target level before you upload anything.

    - **Recommended (AA)** is the default and the best choice for most teams
    - **Advanced (AAA)** is for teams that want the highest coverage level

    The built-in tooltip explains that WCAG levels are cumulative: passing **AA** also requires passing **A**, and passing **AAA** requires passing **A**, **AA**, and **AAA** criteria.
  </Step>
  <Step title="Review the built-in help if needed">
    The **How accessibility testing works** section explains what Uxia is checking and includes FAQ-style help around WCAG levels, legal standards, code-level issues, and accessibility review scope. This is useful for teams that need a quick refresher before launching.
  </Step>
  <Step title="Upload frames or recordings">
    Move to **Upload Frames** and add the screens or videos you want Uxia to analyze. Uxia requires at least one file before you can continue.

    If you click **Next** without uploading anything, the platform shows the validation error **Please upload at least one file**.
  </Step>
  <Step title="Continue to Summary and launch the audit">
    After your files are uploaded, continue to **Summary** to review the setup and launch the test.
  </Step>
</Steps>

## How to choose between AA and AAA

| Level | Best for | What to expect |
| --- | --- | --- |
| **AA** | Most teams, product design reviews, common compliance workflows | The practical baseline. Covers the most common accessibility barriers and aligns with the level most organizations need to target. |
| **AAA** | Inclusive design teams, advanced accessibility programs, stricter internal standards | Higher coverage, but also a stricter bar. Best when you want to go beyond baseline compliance. |

<Note>
  If you do not have a strong reason to target AAA, start with **AA**. That matches the default Uxia recommendation and is the right baseline for most product teams.
</Note>

## What Uxia evaluates

In the live flow, Uxia explains that the Accessibility Test covers areas such as:

- **WCAG 2.2 conformance**
- **Low-vision support**
- **Mobile and touch interactions**
- **Cognitive accessibility**

In practice, that means the report helps teams review issues such as:

- Contrast and non-text contrast
- Text clarity and readability
- Touch target size
- Reading order and visual hierarchy
- Input labels and instructions
- Focus behavior and keyboard-related checks
- Error prevention and consistent navigation

## What the results page shows

A completed Accessibility Test opens into three tabs:

- **Accessibility Results**
- **Vision Simulation**
- **Test Details**

### Accessibility Results

This is the main review tab. It includes:

- A frame selector
- A summary card with **Passed**, **Failed**, and **Review** counts
- Search and filters for **Status**, **Review Scope**, and **Level**
- A criteria table with columns for:
  - **WCAG Code**
  - **Success Criterion**
  - **Level**
  - **Category**
  - **Status**
  - **Review Scope**

Each row can expose:

- A status badge such as **Passed** or **Review**
- A scope label such as **Design** or **Code**
- A **Mark as Passed** action for manual resolution
- A **See More** drawer with the detailed explanation

### Vision Simulation

This tab shows side-by-side previews of the same frame under different visual conditions. In the current UI, the user can compare a baseline view such as **Normal Vision** with a simulated condition such as **Red-Blind**, and adjust severity with sliders.

Use this tab to understand whether contrast, hierarchy, and visual emphasis still work when perception changes.

### Test Details

This tab acts as the audit record. It surfaces:

- The selected conformance level
- Tags
- Uploaded frames
- The accessibility explainer content again for reference

## How to interpret status and scope

### Status definitions

| Status | Meaning |
| --- | --- |
| **Passed** | Uxia considers the criterion satisfied for the current frame or evidence set. |
| **Failed** | A confirmed issue was detected and should be fixed. |
| **Review** | The item needs human judgment, implementation confirmation, or contextual review before it can be cleared. |

### Review scope definitions

| Scope | Meaning |
| --- | --- |
| **Design** | The likely fix belongs in the design system, layout, copy, sizing, hierarchy, or visual treatment. |
| **Code** | The likely fix belongs in implementation, semantics, labeling, keyboard behavior, or accessibility attributes. |
| **Both** | The issue spans design intent and implementation details. |

## What you see inside a criterion drawer

Clicking **See More** opens a right-side detail panel for that WCAG criterion. In the current UI, the drawer includes:

- The criterion code and title
- Status, scope, and WCAG level badges
- A preview of the affected frame
- A plain-language **Description**
- A **How To Verify** section
- A practical **Checklist**
- Quick actions such as **WCAG Criteria** and **Mark as Passed**

This is the most useful view for handoff because it explains not just what is wrong, but how a designer or developer should validate the fix.

## Recommended team workflow

1. Run the test as soon as the target flow is stable enough to review.
2. Fix **Failed** items first.
3. Triage **Review** items with the right owner using the **Design** or **Code** scope labels.
4. Use the criterion drawer as the working checklist.
5. Re-run the test after fixes to confirm progress.

## Important limitation

<Warning>
  Uxia's Accessibility Test helps teams catch issues early, but it is not a substitute for a formal legal accessibility audit or certification. Some WCAG criteria still require expert review, assistive technology testing, and human judgment.
</Warning>
